## 1. Amaç / Purpose

Bu doküman Personel UAM platformunda kullanılan tüm kriptografik primitive'leri,
anahtar hiyerarşilerini, rotation prosedürlerini ve compromise recovery
senaryolarını tek çatı altında toplar. Üçüncü taraf denetim (SOC 2 Type II
cryptographic review, KVKK denetim, ISO 27001 Annex A.10) ve iç güvenlik
revizyonları için referans belgedir.

## 2. Kullanılan Primitive'ler / Primitives in Use

| Primitive | Algoritma | Anahtar Boyutu | Kullanım | FIPS 140-3 | Kaynak Kütüphane |
|---|---|---|---|---|---|
| **AEAD** | AES-256-GCM | 256-bit | Event payload envelope, keystroke content, screen capture at-rest, DPAPI-sealed config | ✅ Approved | `aes-gcm` (RustCrypto) + `crypto/cipher` (Go stdlib) |
| **Key exchange** | X25519 | 256-bit | Enrollment ceremony (agent ↔ control plane) | ⚠️ Approved in FIPS 140-3 rev 2 (SP 800-186) | `x25519-dalek` |
| **KDF** | HKDF-SHA256 | — | PE-DEK derivation, per-session key derivation | ✅ Approved (SP 800-56C) | `hkdf` (RustCrypto) + `crypto/hkdf` |
| **Signing** | Ed25519 | 256-bit | Policy bundle signing, release manifest, audit checkpoint, Evidence Locker items | ⚠️ Approved via SP 800-186 (FIPS 186-5) | `ed25519-dalek` + Vault transit `ed25519` key type |
| **Hashing** | SHA-256 / SHA-384 | — | Audit hash chain, file integrity, cert fingerprint, SBOM hash | ✅ Approved | `sha2` + `crypto/sha256` |
| **Password hashing** | Argon2id | memory=64MiB, t=3, p=4 | Admin local fallback password (OIDC tercih edilir) | ⚠️ Not FIPS; allowed by KVKK | `argon2` (RustCrypto) |
| **Random** | OS CSPRNG | — | Nonce, salt, session ID, token | ✅ Approved | `getrandom` / `crypto/rand` |
| **At-rest DB** | SQLCipher AES-256 | 256-bit | Agent offline queue | ✅ Approved | `rusqlite` with `bundled-sqlcipher` |
| **TLS** | TLS 1.3 | — | Tüm mTLS akışları | ✅ Approved | `rustls` (agent) + Go crypto/tls (gateway/api) |

### 2.1 Nonce Strategy

**AES-GCM nonce reuse** Personel'in en kritik kriptografik failure modu'dur.
Her durumda:

- **Event envelope**: 96-bit nonce = `8-bit version || 64-bit counter || 24-bit random`.
  Counter per-key tek artış; reset yok. Counter overflow öncesi PE-DEK rotation
  tetiklenir (2^64 mesaj — pratikte erişilmez).
- **Screen capture**: Her frame fresh random 96-bit nonce. Duplicate detection
  agent side hash cache ile.
- **DPAPI sealed blob**: 96-bit fresh random per-seal.

**Yasak**: Manuel nonce reuse, deterministic nonce (SIV mode kullanılmıyor
çünkü KVKK amaçları için per-message semantic security yeterli).

### 2.2 X25519 vs P-256

X25519 seçildi çünkü:

1. Constant-time implementations daha olgun (dalek ecosystem).
2. Misuse-resistance daha yüksek (curve equation dışı noktalar reject).
3. NIST SP 800-186 (FIPS 186-5 annex) tarafından approved.

P-256'ya Phase 2'de geçiş: Eğer FIPS 140-3 Level 2 module sertifikasyonu
gerekirse (kurumsal müşteri talebi), hybrid X25519 + P-256 ECDH'e geçilir.

### 3.1 Anahtar ömürleri / Key lifetimes

| Anahtar | Lifetime | Rotation Trigger | Rotation Mechanism |
|---|---|---|---|
| Control Plane Root Key | 5 yıl | Takvim + compromise | Vault `transit/keys/control-plane/rotate` + rekey all signatures |
| TMK (per tenant) | 2 yıl | Takvim + compromise | Vault `transit/keys/tmk-{tenant}/rotate` + re-derive all PE-DEKs |

| Agent cert | 90 gün | 60. günde auto-renew | `enroll.exe --renew` veya `personel-agent renew` |

| Anahtar | Soğuk depolama / Cold Storage | Sıcak depolama / Hot Storage |
|---|---|---|
| Control Plane Root | Vault Shamir 3-of-5 seal key + encrypted backup USB (air-gapped safe) | Vault memory (unsealed) |
| TMK | Same | Vault transit engine |
| PE-DEK | — (generated on demand) | Postgres wrapped with TMK (wrapped ciphertext) |
| Agent private key | DPAPI LocalMachine scope (Windows CNG) | Agent process memory (zeroed on exit) |
| Gateway private key | TLS file, 0600 perms, root:root, at-rest encrypted volume | Gateway process memory |
| PKI root priv | Offline HSM (YubiHSM2 veya Ledger) — Phase 2 | Not loaded in hot path |

## 4. Keystroke İçerik Şifreleme Akışı / Keystroke Content Encryption Flow

**ADR 0009 + ADR 0013 zorunluluk**: Admin, herhangi bir API veya rol ile
keystroke ciphertext'ini çözemez. Bu matematik bir invariant; "erişim kontrolü"
değil.

### 4.1 Default state (DLP off)

```
Agent capture → [policy check: content_enabled?]
                      │
                      ├── false (default) → KeystrokeWindowStats only
                      │                      (counts, no content)
                      │
                      └── true (opted-in)  → See 4.2
```

### 4.2 Opted-in state

```
Agent capture → AES-256-GCM encrypt(plaintext, PE-DEK, fresh nonce)
             → Upload ciphertext to MinIO sensitive/keystroke/
             → Emit event with ciphertext MinIO key

DLP service:
  1. Derive DSEK from Vault transit derive(TMK, endpoint_id)
     ↳ Requires valid Vault token (requires AppRole Secret ID)
  2. Unwrap PE-DEK with DSEK
  3. Decrypt ciphertext in isolated memory
  4. Run regex/ML classifier (Turkish TCKN, IBAN, secrets)
  5. Emit dlp.match event (redacted metadata only: rule_id, severity)
  6. Zero plaintext + PE-DEK + DSEK immediately

Admin console:
  - Reads dlp.match events only (regex/classifier hits, no content)
  - Has NO derive permission on Vault
  - Has NO PE-DEK unwrap path
  - Cryptographic impossibility, not ACL
```

### 4.3 Vault audit proof

Vault audit device her `transit/derive/tmk-{tenant}` çağrısını logluyor.
Default state'te bu log **boş**. Opted-in state'te sadece `dlp-service`
AppRole token'ı görünüyor.

**Legal claim**: "Bu deployment'ta keystroke decryption anahtarı hiçbir
sürece ulaşmıyor. Vault audit cihazı sıfır `derive` çağrısı kanıtlıyor.
Admin keystroke içeriğine erişim matematik olarak imkansız çünkü decryption
yolu mevcut değil."

## 5. Rotation Programı / Rotation Schedule

### 5.1 Takvim bazlı / Calendar-based

| Interval | Rotation | Owner |
|---|---|---|
| Günlük | Audit checkpoint signature (prev-day hash chain) | Auto (audit-checkpoints cron) |
| Haftalık | Gateway → agent mTLS session re-key handshake | Auto (ticket lifetime TLS 1.3) |
| 60 gün | Agent leaf cert renewal | Auto (enroll --renew) |
| 90 gün | Gateway + service leaf cert renewal | Auto (cert-rotator systemd timer) |
| 1 yıl | PE-DEK rotation | Manuel DPO + ceremony (Phase 2 auto) |
| 2 yıl | TMK rotation | Manuel DPO ceremony |
| 2 yıl | PKI intermediate rotation | DPO + Security ceremony |
| 5 yıl | Control Plane Root Key rotation | Executive + DPO + Security |
| 10 yıl | PKI root CA rotation | Full air-gapped ceremony |

### 5.2 Event-driven rotation

Aşağıdaki olaylar, zamanlamadan bağımsız, **anında** rotation'ı tetikler:

1. **Key compromise suspected** → affected key + downstream rotation
2. **Personnel departure** (DPO, Security lead, Ops manager) → ceremony keys
3. **Vault unseal key loss** → re-initialize + restore from backup
4. **HSM hardware failure** → switch to backup HSM + reissue
5. **CVE in crypto library** → library upgrade + audit affected flows
6. **Audit finding** → per-finding action

## 6. Compromise Recovery Prosedürleri

### 6.1 Agent cert compromise

1. **Detect**: Gateway sees duplicate cert usage from disparate IPs veya
   anti-tamper alert from agent.
2. **Contain**: Add cert serial to Vault PKI CRL; gateway refreshes CRL
   cache < 5 min.
3. **Evict**: Mark endpoint `revoked=true` in Postgres; agent next
   handshake fails; operator ticket fires.
4. **Reissue**: Customer ops regenerates enroll token + re-enrolls machine.
5. **Audit**: Emit `endpoint.cert_revoked` + `KindPrivilegedAccessSession`
   if breach confirmed.

### 6.2 TMK compromise

TMK plaintext leak = tenant-wide catastrophic:

1. **Contain**: Vault `transit/keys/tmk-{tenant}/config` → `deletion_allowed=true`,
   then rotate (new version v+1).
2. **Re-derive**: All PE-DEKs for this tenant re-generated from new TMK version.
3. **Re-wrap**: All stored ciphertext re-encrypted offline (one-time batch).
   Phase 2 automated; Phase 1 manuel script `infra/scripts/tmk-rewrap.sh`
   (AWAITING: script not yet implemented — see §10).
4. **Invalidate old version**: Vault `transit/keys/tmk-{tenant}/config`
   → `min_decryption_version=<new>`.
5. **Audit**: Executive + DPO + Legal notification; KVKK Veri İhlali
   Bildirimi 72h SLA.

### 6.3 Control Plane Root Key compromise

Worst case. All signed artifacts are untrustworthy:

1. **Contain**: Freeze policy push, release pipeline, audit checkpoint.
2. **Ceremony**: Executive + DPO + Security 3-of-5 unseal → generate new
   root → backup Shamir shares.
3. **Re-sign**: All historical policy bundles, audit checkpoints,
   release manifests re-signed with new root.
4. **Verify**: Agents reject old signatures on next policy pull; operators
   push new trust anchors via `enroll --rotate-trust`.
5. **Audit**: Full incident report per SOC 2 CC7.3; 72h KVKK bildirim.

### 6.4 Vault seal key compromise

3-of-5 Shamir threshold: 3 shares leaked = seal keys compromised but
Vault unseal itself not compromised unless Vault data ALSO leaked:

1. **Re-key**: `vault operator rekey` → new Shamir shares distributed.
2. **Rotate underlying master**: `vault operator rotate` (rotates data
   encryption key).
3. **Audit**: Which shares leaked? Rotate affected ceremony participants.

## 7. FIPS 140-3 Uyum Notları / Alignment Notes

Personel Faz 1'de **FIPS 140-3 validated module kullanmıyor** ancak tüm
seçilen algoritmalar FIPS 140-3 approved veya NIST SP 800-186 approved.

**Faz 2 hedef**: BoringCrypto (Go) + `rustls-fips` (Rust) kütüphane
geçişi kurumsal müşteri talebine göre. ClickHouse + Vault Enterprise
FIPS modları değerlendirilecek.

**Gap**:
- Argon2id FIPS approved değil → admin lokal fallback parola hashing için.
  Mitigation: OIDC (Keycloak) tercihli; lokal password sadece break-glass.
- X25519 FIPS 186-5 / SP 800-186 approved ama 140-3 module test suite
  kapsama alanı henüz sınırlı. Alternatif: P-256 ECDH.

## 9. Reviewer Checklist / Denetçi Kontrol Listesi

Yeni bir cryptographic feature PR review ederken:

- [ ] Yeni primitive eklendi mi? Varsa §2 tablosuna ekle + FIPS status.
- [ ] Nonce strategy doğrulandı mı? AES-GCM için unique nonce garantisi var mı?
- [ ] KDF girdi salt değeri unique + rastgele mi?
- [ ] Yeni anahtar hiyerarşisi §3 diagramına yansıtıldı mı?
- [ ] Rotation schedule §5 güncellendi mi?
- [ ] Compromise recovery senaryosu §6 güncellendi mi?
- [ ] Unit test constant-time comparison için `subtle` crate kullanıyor mu?
- [ ] Anahtar memory'de sonlandırılınca zeroize ediliyor mu (`zeroize` trait)?
- [ ] Plaintext anahtar log'a yazılıyor mu? (Kesinlikle hayır.)
- [ ] Test fixture'lar gerçek anahtar içeriyor mu? (Kesinlikle hayır; `testdata/`
      altı gitleaks allowlist'inde olmalı + deterministic test key kullanılmalı.)
- [ ] KVKK m.6 özel nitelikli veri için ayrı encryption zone gerekli mi?
- [ ] ADR 0013 invariant bozuluyor mu? (DLP off iken derive path oluşturulmaz.)
- [ ] `personel-crypto` crate public API'de key material leak ediliyor mu?
- [ ] FIPS-only build tag (`// +build fips`) gerekliyse eklendi mi?
- [ ] Threat model (`docs/security/threat-model.md`) güncellendi mi?
