# Key Hierarchy — Keystroke Content Encryption

> Language: English. Status: LOCKED DESIGN. This document enforces Decision 4: admins cannot see raw keystrokes, by construction. Amended by **ADR 0013 — DLP Disabled by Default in Phase 1** (2026-04-11): the cryptographic structure below is unchanged, but the **default runtime state** now has no process capable of deriving DSEK. See §Default vs Opted-In Runtime Guarantees.

## Default vs Opted-In Runtime Guarantees (ADR 0013)

The cryptographic hierarchy, the Vault policies, and the code that handles keystroke content are identical in both states. What differs is whether any process is authorised, at a given moment, to walk from TMK → DSEK → PE-DEK → plaintext.

| Property | Default state (DLP off) | Opted-in state (DLP on) |
|---|---|---|
| `personel-dlp` container running? | **No.** `profiles: [dlp]` in docker-compose; not started. | Yes, per Profile 1 or Profile 2 in `dlp-deployment-profiles.md`. |
| `dlp-service` Vault policy exists? | Yes (created at install for audit review). | Yes (unchanged). |
| `dlp-service` AppRole Role ID exists? | Yes. | Yes. |
| `dlp-service` AppRole **Secret ID** issued? | **No.** Without a Secret ID, no login, no token, no derive. | Yes, single-use, issued by `infra/scripts/dlp-enable.sh` after sign-off verification. |
| Vault `derive` call count over the deployment's lifetime (so far) | **Zero.** Provable from Vault audit device. | ≥ 1, beginning at the timestamp of the opt-in ceremony. |
| Is DSEK materialised in any process memory? | **No.** DSEK requires TMK `derive`, which requires a Vault token, which requires a Secret ID, which has never been issued. | Yes, in the DLP service process memory, subject to zeroization and rotation rules below. |
| Are wrapped PE-DEKs stored in Postgres? | Yes, **empty table** until DLP is enabled (PE-DEK generation happens at enrollment **by the DLP service** — see §Hierarchy). In the default state, no PE-DEKs exist. | Yes, populated as endpoints enroll after opt-in. |
| Is keystroke content captured by the agent? | Yes (if policy `keystroke.content_enabled = true`). | Yes. |
| Is keystroke content encrypted by the agent? | **Partially — see note below.** | Yes, with PE-DEK, AES-256-GCM. |
| Is keystroke ciphertext uploaded to MinIO `sensitive/keystroke/`? | See note below. | Yes. |
| Is keystroke ciphertext ever decrypted? | **No.** No derivation path exists. | Yes, inside the DLP service for pattern matching. |
| Sensitive-flagged TTL honoured? | Yes (14 days). Ciphertext (if any) is destroyed on schedule even though it is never read. | Yes. |
| Legal claim | "No process holds or derives the keystroke decryption key. Vault audit proves zero `derive` calls. Admin access to keystroke content is mathematically impossible in this deployment because no decryption path exists." | "Only the isolated DLP engine can decrypt, and only for pattern matching producing redacted `dlp.match` metadata. Admins cannot read raw keystrokes, by construction." (unchanged) |

**Important note — PE-DEK existence in the default state**: The `§Hierarchy` section below states that PE-DEK is generated by the DLP service at endpoint enrollment. Because the DLP service does not run in the default state, **no PE-DEK is ever generated**, which means the agent has no key with which to encrypt keystroke content. Two operational consequences flow from this, and downstream agents must honour them:

1. In the default state, the Phase 1 expectation is that **the agent's `keystroke_content` collector is disabled at policy level** (`KeystrokeSettings.content_enabled = false` in the default policy bundle for any tenant that has not opted in to DLP). The agent still collects `KeystrokeWindowStats` (counts, not content). This is the second-level opt-out noted in ADR 0013 Consequences §Negative and tracked as part of exit criterion #18.
2. If a customer opts in later, the DLP enable script (`infra/scripts/dlp-enable.sh`) is responsible for: (a) generating PE-DEKs for all already-enrolled endpoints, (b) delivering them to the agents via the sealed enrollment channel on next stream open using the `RotateCert` → rekey path, and (c) flipping the `KeystrokeSettings.content_enabled` flag in the policy bundle. Until that script completes, no keystroke content is captured.

Put differently: the default state is not merely "DLP is dormant." It is "the keystroke decryption key never comes into existence on this deployment," and the agent-side content collector is wired to respect that fact.

### Runtime Guarantees in Default State — Proof Sketch

- **Vault audit device** records zero `transit/keys/tenant/*/tmk` derive operations over the deployment's lifetime. This is a positive, auditable fact, not an absence-of-evidence argument.
- **Vault** records zero `auth/approle/login` calls for the `dlp-service` role, because no Secret ID has been issued.
- **Postgres** `keystroke_keys` table is empty.
- **MinIO** `sensitive/keystroke/` prefix is either empty or contains nothing (because the agent collector is disabled per consequence 1 above). If a customer's policy has been manually changed to enable content capture without enabling DLP — which the API refuses to do per the state endpoint invariant — the ciphertext would exist but would be TTL-destroyed in 14 days and would never be readable by any process.
- **Admin Console** header badge shows `DLP: Kapalı / DLP: Disabled`. The state endpoint `GET /api/v1/system/dlp-state` is the single source of truth for this badge and is cross-validated at every request against the presence of the `personel-dlp` process, the existence of an active AppRole Secret ID in Vault, and the most recent `dlp.enabled|disabled` audit chain event.

### Runtime Guarantees in Opted-In State

Unchanged from the pre-ADR-0013 design. See §Why Admins Cannot Decrypt below and §Trust Model: Two Deployment Profiles further down.

## Threat Model (Specific to Keystroke Content)

Adversaries considered:

1. A malicious **platform admin** (console user with full tenant rights) attempting to view raw keystrokes.
2. A compromised **Admin API** or database trying to decrypt blobs.
3. A compromised **Ingest Gateway**.
4. A compromised **endpoint** (already assumes full loss of that endpoint's keys and plaintext; we do not defend past this for that one machine).
5. An insider **operator** with filesystem access to Vault storage.

Goal: under threats 1–4, raw keystroke content is cryptographically inaccessible. Only the **DLP Service** — running in a separate trust zone — can decrypt, and even then only for pattern-matching purposes producing non-content metadata.

Non-goal: hiding the existence of keystroke blobs, their lengths, timestamps, or the endpoint they came from.

## Hierarchy

```
Personel Keystroke Crypto Root (KCR)
  - stored in Vault KV v2 at kv/crypto/kcr (SEALED; Shamir split 3-of-5)
  - never exported; used only to bootstrap TMKs
  |
  +-> Tenant Master Key (TMK) — one per tenant
  |     - Vault transit engine: transit/keys/tenant/<tenant_id>/tmk
  |     - exportable: false
  |     - allowed operations: derive (HKDF-SHA256)
  |     - only identity allowed to derive: DLP Service (Vault policy dlp-service)
  |
  +-> DLP Service Encryption Key (DSEK) = HKDF(TMK, info="dlp-service-v1")
  |     - materialized only inside DLP Service memory
  |     - never written to disk
  |     - rotated on DLP service restart and on schedule (24 h)
  |
  +-> Per-Endpoint Data Encryption Key (PE-DEK)
        - generated fresh at endpoint enrollment by the DLP Service
        - PE-DEK plaintext is wrapped with DSEK via AES-256-GCM
        - wrapped form stored in Postgres: keystroke_keys(endpoint_id, wrapped_dek, nonce, created_at, version)
        - the plaintext PE-DEK is delivered to the endpoint ONCE via an ephemeral X25519-sealed channel bound to the agent's enrollment public key
        - the endpoint stores PE-DEK in a DPAPI-protected blob sealed to TPM when available
```

### Keystroke Encryption at the Endpoint

1. Collector gathers keystroke content per active window into a buffer (bounded, per-window).
2. At flush, encrypt: `ciphertext = AES-256-GCM(key=PE-DEK, nonce=random96, aad=endpoint_id||seq, plaintext=buffer)`.
3. Upload ciphertext blob to MinIO path `keystroke-blobs/<tenant>/<endpoint>/<date>/<ulid>.bin`.
4. Emit `keystroke.content_encrypted` metadata event referencing the blob, nonce, AAD, and `key_version`. **No plaintext leaves the endpoint.**

### Keystroke Decryption in DLP Service

1. DLP subscribes to NATS subject `events.v1.keystroke.content_encrypted`.
2. For each event: look up `wrapped_dek` for `endpoint_id` from Postgres.
3. Unwrap with DSEK (in-memory) to get PE-DEK (in-memory only).
4. Fetch blob from MinIO.
5. AES-256-GCM decrypt with `PE-DEK`, nonce, AAD.
6. Run plaintext through DLP rule engine (regexes, TCKN, IBAN, credit card Luhn, custom tenant rules).
7. Emit `dlp.match` events containing **only**: rule id, severity, window exe, endpoint, occurrence count, a **redacted** snippet (e.g., `TCKN: ***`), and a policy-configurable short hash of the matched region.
8. **Zeroize** plaintext buffer, PE-DEK working copy, GCM tag buffer.
9. **DLP Service never writes plaintext or PE-DEK to disk or to any outbound RPC other than `dlp.match`.**

## Why Admins Cannot Decrypt (Proof Sketch)

- Admin console identity has **zero** Vault policy granting access to `transit/keys/tenant/*/tmk` derive operations or `kv/crypto/kcr`.
- `keystroke_keys` in Postgres contains only **wrapped** DEKs. Wrap key (DSEK) is not reconstructable without TMK derive capability.
- TMK is non-exportable in Vault transit; Vault returns only the result of derive operations, and only to the `dlp-service` AppRole.
- Admin API has no RPC path in `proto/personel/v1` that returns a decrypted blob or an unwrapped key. The `dlp` proto surface is strictly `dlp.match` metadata.
- The Vault audit device records every TMK derive call; unexpected derives alert the DPO.

If the threat model changes (e.g., compliance requires admins to see raw content), it would require a deliberate, audited code change + Vault policy change + new proto — i.e., it is visibly non-accidental.

## Trust Model: Two Deployment Profiles

The isolation requirements below are the **minimum** applicable to any deployment. The full trust model operates in two tiers, authoritatively defined in `dlp-deployment-profiles.md`:

- **Profile 1 — Hardened Container (Phase 1 default shape *when opted in*)**: DLP runs as a distroless, read-only, seccomp+AppArmor-confined, mlockall'd, nftables-egress-restricted container on the shared application host. Legal claim: "admins cannot read keystrokes under the standard deployment's enumerated hardening assumptions." Per ADR 0013, this profile applies only after the customer has completed the opt-in ceremony; until then, no DLP container is running and the stronger "key never existed" claim applies (§Default vs Opted-In Runtime Guarantees above).
- **Profile 2 — Dedicated Host (Phase 2 default shape *when opted in*, GA for strict customers)**: DLP runs on a dedicated VM or physical node classified as a high-value asset by the customer DPO, with no other Personel workloads co-resident. Legal claim: "admins cannot read keystrokes, by construction." Same ADR 0013 caveat: applies only after opt-in.

The choice of profile is recorded at install time, referenced in the customer contract addendum, and re-verified by the annual red-team exercise (Phase 1 exit criterion #9). Phase 1 pilot customers accept Profile 1; Phase 2 GA defaults to Profile 2 for any regulated-sector customer.

## Rotation

- **TMK**: rotated annually via Vault transit rotate. Old version retained for decrypt of legacy blobs (configurable grace 30 d). After grace, old blobs are re-wrapped by DLP service batch job or, if expired per retention, purged.
- **DSEK**: derived fresh on every DLP restart and every 24 h; wraps are versioned so older wrapped DEKs unwrap cleanly.
- **PE-DEK**: rotated on suspected endpoint compromise, on hardware fingerprint mismatch, and on demand by DPO.

## Destruction = Key Destruction

When retention expires for a keystroke blob:
1. Vault transit TMK version used to wrap that endpoint's DEK is scheduled for destruction after all blobs wrapped under it have expired.
2. PE-DEK wrap record in Postgres is deleted.
3. MinIO lifecycle deletes the ciphertext blob.
4. Because ciphertext + wrapped key + TMK version are all gone, the data is cryptographically unrecoverable. Audit log records the destruction with the TMK version id.

## Key Version Handshake (Stream Open)

Every agent gRPC stream begins with `AgentMessage.Hello`. To detect stale or rotated keys **before** any event is accepted, `Hello` carries two version fields (see `proto/personel/v1/agent.proto`):

- `uint32 pe_dek_version` — the PE-DEK generation this endpoint currently holds.
- `uint32 tmk_version` — the TMK generation under which this endpoint's PE-DEK was last wrapped (tenant-scoped).

Gateway handshake logic:

1. Look up the expected `(pe_dek_version, tmk_version)` in Postgres `keystroke_keys` for this endpoint.
2. If `Hello.tmk_version` < current TMK generation for the tenant (TMK has been rotated while the endpoint was offline): refuse `EventBatch` acceptance, send `ServerMessage.RotateCert` with `reason="rekey"` and initiate the re-enrollment path. Queue on the agent side is retained; no data is lost.
3. If `Hello.pe_dek_version` < expected PE-DEK version (DPO forced PE-DEK rotation, e.g., suspected endpoint compromise): same refuse-and-rekey flow.
4. If the endpoint is ahead of the server (impossible under correct operation): log as `agent.tamper_detected` with severity HIGH; alert.
5. If versions match: accept `EventBatch` stream.

The gateway **never** decrypts; it validates versions by comparing integers. This keeps the admin plane zero-knowledge of any key material while still catching replay or rollback attempts before ingestion.

Version negotiation is purely forward — the server only tells the agent "rotate," never "downgrade." This is enforced at the proto level: `ServerMessage` has no "use older key" payload.

## Code-Level Rules for Downstream Agents

- **backend-developer**: MUST NOT add any RPC returning decrypted keystroke content. CI linter rule (to be authored) greps for `keystroke` + `plaintext` in proto and fails.
- **rust-engineer**: Agent MUST zeroize keystroke buffers post-encrypt. PE-DEK MUST NOT be logged, serialized, or swapped to disk unprotected.
- **security-engineer**: Owns DLP service policy, Vault policies, and the annual red-team test proving admins cannot access raw content.
- **postgres-pro**: `keystroke_keys` table has no index on wrap material, no backup exported in plaintext; backups are encrypted at the pg_basebackup layer.
- **nextjs-developer**: Console must have NO component that renders keystroke content. Only `dlp.match` DTOs.
